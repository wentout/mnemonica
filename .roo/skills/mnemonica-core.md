# Mnemonica Core - AI Skill

## Overview

Mnemonica is an instance inheritance system for JavaScript/TypeScript. It enables prototype chain-based type definitions through the `define()` function, creating explicit inheritance graphs that eliminate common prototype bugs.

## Core Concepts

### The `define()` Function
```typescript
const MyType = define('MyType', function (this: MyType, data: Data) {
	Object.assign(this, data);
});
```

Returns a constructor with methods:
- `.define()` - define subtypes
- `.lookup()` - find types by path  
- `.registerHook()` - lifecycle hooks

### Instance Inheritance
Instances inherit through prototype chains:
```typescript
const SubType = MyType.define('SubType', function (this: SubType, extra: string) {
	this.extra = extra;
});
```

### Type-Safe Lookup with `lookupTyped()`

**What is Tactica?** `@mnemonica/tactica` is a companion TypeScript Language Service Plugin that analyzes your source files and generates type definitions for Mnemonica's dynamic constructors. It lives in a separate directory (`tactica/` in the monorepo) and is published as `@mnemonica/tactica` on npm.

Tactica generates `.tactica/types.ts` files with TypeRegistry augmentation:

```typescript
// Generated by tactica - do not edit manually
declare module 'mnemonica' {
	interface TypeRegistry {
		'UserType': new (...args: unknown[]) => UserTypeInstance;
		'UserType.AdminType': new (...args: unknown[]) => AdminTypeInstance;
	}
}
```

Then `lookupTyped()` provides compile-time type safety:
```typescript
import { lookupTyped } from 'mnemonica';

// Fully typed constructor
const UserType = lookupTyped('UserType');
const user = new UserType({ name: 'John' }); // TypeScript knows the signature
```

Without tactica: use `lookup()` (returns `unknown`, runtime only)
With tactica: use `lookupTyped()` (returns typed constructor, compile-time safety)

### Key Symbols
- `SymbolConstructorName` - type name on constructors
- `SymbolParentType` - parent type reference
- `SymbolSubTypes` - subtypes map

## Critical Rules

### Code Style
- **Tabs only** for indentation (width 4)
- **Space before function parentheses**: `function myFunc () {}`
- **Space after colons**: `{ key: value }`
- TypeScript `strict: true`, no unused variables

### Type vs Interface
- **Type** for instance data: `type UserData = { name: string }`
- **Interface** only for behavior contracts (constructors, methods)

### Testing Requirements
- **100% coverage** required (statements, branches, functions, lines)
- Run `npm run test:cov` before completing any task
- Run `npm run test:jest:cov` for Jest coverage
- Mocha tests run on transpiled `build/` code
- Jest tests run on TypeScript `src/` directly

## Architecture

### File Organization
```
src/
├── index.ts           # Main exports: define, lookup, apply, call, bind
├── types/index.ts     # TypeScript definitions
├── constants/         # Symbols, default options
├── descriptors/       # Type collections, error definitions
├── api/               # Core implementation
│   ├── types/         # TypeProxy, InstanceCreator, Mnemosyne
│   ├── errors/        # Error handling
│   ├── hooks/         # Lifecycle hooks
│   └── utils/         # Utilities
└── utils/             # Public utilities (extract, parse, merge)
```

### Proxy-Based Architecture
- **TypeProxy** - Wraps type constructors
- **Mnemosyne** - Instance prototype handler
- **TypesCollection** - Registry for types

## Common Tasks

### Adding a New Type Definition
1. Define interface in `src/types/index.ts`
2. Update `MnemonicaInstance` interface if needed
3. Add tests in both Mocha and Jest
4. Run `npm run test:cov`

### Adding a Hook
```typescript
MyType.registerHook('postCreation', (opts) => {
	// opts.instance, opts.type, opts.args
});
```

### Error Handling
```typescript
import { constructError } from '../api/errors/index.js';
const MyError = constructError('MY_ERROR', 'Error message');
throw new MyError('additional info', stack);
```

## Pitfalls to Avoid

1. **Never use assignment for prototypes** - always `Object.setPrototypeOf`
2. **Don't reuse constructors directly** - creates corrupted chains
3. **Always maintain 100% test coverage** - no exceptions
4. **Never modify tsconfig.json or eslint.config.js** without approval
5. **No timing-based code for UI interactions** - use native events only
