# How `lookupTyped` works (and where tactica fits)

> The runtime Trie that mnemonica builds is invisible to TypeScript by default. The fix is one TypeScript-level pattern: declaration-merge mnemonica's `TypeRegistry` interface so that `lookupTyped(path)` resolves to a fully-typed constructor. The augmentation can be hand-written or generated by [`@mnemonica/tactica`](https://www.npmjs.com/package/@mnemonica/tactica). The runtime behaviour is identical either way.
>
> For a side-by-side recipe of both paths, see [`./typed-lookup.md`](./typed-lookup.md).

---

## The problem in one sentence

mnemonica builds a Trie of constructors at runtime. TypeScript has no idea the Trie exists. `lookupTyped` + an augmented `TypeRegistry` tells TypeScript.

Without that bridge, every chain operation requires a cast.

---

## What goes wrong without the bridge

```typescript
import { RequestData } from './collections/requestTypes.js';

const requestData = new RequestData({ method: 'GET', url: '/' });
const routeData   = new requestData.RouteData({ pagePath: '/' });
// Error: Property 'RouteData' does not exist on type 'RequestData'
```

TypeScript rejects valid runtime code. You are forced to write:

```typescript
const routeData = new (requestData as any).RouteData({ ... });
// or worse:
const routeData = new requestData.RouteData({ ... }) as unknown as RouteDataT;
```

This is not type safety. This is type suppression.

---

## What the bridge looks like

```typescript
import { lookupTyped } from 'mnemonica';

const RequestData = lookupTyped('RequestData');

const requestData = new RequestData({ method: 'GET', url: '/' });
const routeData   = new requestData.RouteData({ pagePath: '/' });
const pageData    = new routeData.PageData({ header: {...}, content: '' });
```

Zero casts. Zero `any`. Zero `unknown`. TypeScript knows every property, every constructor, every link in the chain.

For this to work, mnemonica's `TypeRegistry` interface needs to be augmented with your types via TypeScript declaration merging.

---

## Two ways to produce the augmentation

### By hand

For small projects or for learning, you can write the augmentation directly in a `.d.ts` (or `.ts`) file:

```typescript
// types/mnemonica-registry.d.ts
import type { ProtoFlat } from 'mnemonica';

export type RequestDataInstance = {
    method    : string;
    url       : string;
    RouteData : new (data: { pagePath: string }) => RouteDataInstance;
};

export type RouteDataInstance = ProtoFlat<RequestDataInstance, {
    pagePath  : string;
    PageData  : new (data: { header: object; content: string }) => PageDataInstance;
    RouteData : undefined;
}>;

declare module 'mnemonica' {
    interface TypeRegistry {
        'RequestData'           : new (data: { method: string; url: string }) => RequestDataInstance;
        'RequestData.RouteData' : new (data: { pagePath: string }) => RouteDataInstance;
    }
}
```

Include the file in `tsconfig.json` `include`. You are done.

### Via tactica

For projects with many types or for hands-off maintenance, install [`@mnemonica/tactica`](https://www.npmjs.com/package/@mnemonica/tactica) and run it as a CLI/codegen utility. It scans your `define()` calls and generates the augmentation:

```bash
npm install --save-dev @mnemonica/tactica
npx tactica --output ./.tactica --include "src/**/*.ts"
```

Include the generated files in `tsconfig.json`:

```json
{
    "include": ["src/**/*.ts", ".tactica/**/*.ts"]
}
```

Tactica writes the same shape of augmentation a careful human would, using `ProtoFlat<Parent, ...>` for nested types and filling in instance-level subtype constructors automatically. The runtime is unaffected. Re-run the CLI whenever your type graph changes, or wire it into your build script.

See [`./typed-lookup.md`](./typed-lookup.md) for a side-by-side recipe with both paths.

---

## The runtime is identical

`lookupTyped('RequestData')` and `import { RequestData }` return the **same constructor object** at runtime. The only difference is what TypeScript sees at compile time:

| Approach | Runtime | Compile-time |
|---|---|---|
| Direct import without augmentation | ✅ Constructor works | ❌ TypeScript blind to sub-types |
| `lookupTyped` with augmentation (manual) | ✅ Same constructor | ✅ TypeScript knows the Trie |
| `lookupTyped` with augmentation (tactica) | ✅ Same constructor | ✅ TypeScript knows the Trie |

There is no runtime cost. There is no indirection. There is only **type safety, conditional on the augmentation being present**.

---

## The rule

The anti-patterns are the same whether you augment by hand or via tactica:

- Do not write `as unknown as` casts on mnemonica types.
- Do not import constructors directly from collection files when `lookupTyped` would work.
- Do not accept "it compiles with casts" as good enough.

What changes is how you keep the `TypeRegistry` augmentation in sync with your code:

- **Hand-written:** edit the `.d.ts` when types change.
- **Tactica:** the CLI/codegen utility keeps `.tactica/` in sync.

Pick whichever fits your project. Both are first-class.
